# 文墨（Wenmo）—— 项目概括文件

> 版本：v1.1  
> 定位：让 AI 写出有温度、有性格、像真人一样的「人话」  
> 本文档在原始需求基础上做了结构化整理与适度架构调整，作为团队对齐、开发排期与验收的统一参考。  
> **v1.1**：主数据库改为云端托管 PostgreSQL（厂商待定），见 4.4 节。

---

## 一、项目概述

### 1.1 一句话定义

**文墨**是一款以「去 AI 味」和「人格化」为核心竞争力的 AI 写作与沟通助手。它不追求「正确但空洞」的标准答案，而是帮助用户产出**读起来像自己写的、聊起来像朋友在说话**的内容。

### 1.2 核心问题

当前 AI 写作工具的通病：

| 现象 | 根因 |
|------|------|
| 满篇「赋能」「抓手」「深度洞察」 | 指令模糊，模型只能调用最「标准」的书面语模板 |
| 语法正确但没人味儿 | 缺少明确的语气、身份、受众与风格约束 |
| 每次对话都像从零开始的客服 | 缺少连贯人格与上下文记忆 |
| 中文一词多义、风格差异大 | 过于「完美」的输出反而暴露机器感 |

**结论**：问题不在 AI 能力本身，而在于**如何把人的风格、场景和人格，工程化地注入生成过程**。

### 1.3 产品愿景

- **短期**：用户上传自己的文案样本，生成「像自己写的」内容；在特定沟通场景下，AI 能替用户「像朋友一样聊天」。
- **中期**：建立可复用的「风格档案」与「数字人格」，跨模块共享，越用越像用户。
- **长期**：成为中文场景下「拟人化写作」的标杆产品，在博客、社媒、商务沟通、内容审核等场景形成差异化。

### 1.4 目标用户

| 用户类型 | 典型场景 | 核心诉求 |
|----------|----------|----------|
| 内容创作者 | 公众号、小红书、短视频脚本 | 保持个人文风，批量产出但不失「人味」 |
| 职场写作者 | 报告、邮件、方案、合同审阅 | 专业但不生硬，能识别文案风险 |
| 普通用户 | 微信聊天、邀约、说服、安慰 | 帮我想怎么说，而不是帮我写「官方回复」 |
| 法务/合规相关 | 口供、合同、政策文本 | 快速定位潜在漏洞与风险表述 |

### 1.5 核心竞争力（与竞品的差异）

1. **去 AI 味**：从提示词工程、后处理检测到风格约束，全链路对抗「机器腔」。
2. **风格克隆**：3–5 篇样本即可提取用户笔风，生成内容带个人用词与句式习惯。
3. **人格化对话**：不是问答机器人，而是有语气、有情绪、有策略的沟通伙伴。
4. **可工程化的提示词体系**：模板版本化、A/B 测试、审核机制，质量可持续迭代。

## 二、核心理念：三大支柱

### 2.1 支柱一：风格学习与迁移（从模仿到内化）

**目标**：让系统不仅按通用规则写「人话」，还能学习特定用户风格，实现「千人千面」。

**流程**：

```
用户上传 3–5 篇历史文案
    ↓
样本分析（用词、句长、修辞、语气词、标点习惯）
    ↓
风格向量化（embedding / 特征向量）
    ↓
生成时注入风格约束 + 一致性校验
    ↓
输出「像用户自己写的」文案
```

**关键指标**：

- 风格相似度（与用户样本的余弦相似度）≥ 0.75（可调）
- 用户主观评分「像我自己写的」≥ 4/5

### 2.2 支柱二：人格化对话（从应答到交流）

**目标**：聊天模块像朋友，不像客服。

**设计原则**：

- 允许口语、语气词、适度「不完美」
- 感知对方情绪，调整回应策略（安慰 / 调侃 / 说服 / 转移话题）
- 跨轮次保持人设一致，避免前后矛盾
- 明确角色关系（朋友、同事、客户等）与对话目标（邀约、说服、道歉等）

**反例 vs 正例**：

| 场景 | ❌ AI 味 | ✅ 人话 |
|------|---------|--------|
| 朋友说「今天好累」 | 「我理解您的疲惫，建议您适当休息。」 | 「累啥啊，打两把就精神了，快来！」 |
| 邀请聚餐 | 「诚挚邀请您参加本次聚餐活动。」 | 「周五晚上老地方，来不来？不来我点你外卖。」 |

### 2.3 支柱三：场景化智能（从通用到专用）

**目标**：不同场景用不同「大脑配置」——检索、写作、聊天各自有独立的提示词模板、后处理规则与评估标准，但共享用户风格档案与人格设定。

## 三、功能模块详解

### 3.1 模块一：文案智能检索

#### 功能描述

用户上传文案（口供、合同、报告、政策文件等），系统自动检索并标注：

- 潜在逻辑漏洞
- 违法或合规风险点
- 表述歧义与模糊条款
- 前后矛盾之处

输出形式：**结构化风险列表 + 自然语言解读报告**（后者必须「说人话」，不能是条目堆砌）。

#### 用户流程

1. 上传文本 / 粘贴内容 / 上传 PDF（二期）
2. 选择文档类型（合同 / 报告 / 口供 / 通用）
3. 提交分析任务，获得任务 ID
4. 查看风险热力标注 + 分级报告（高 / 中 / 低）
5. 导出报告（Markdown / PDF）

#### 技术要点

- **语义理解 + 风险识别**：自研深度学习模型（PyTorch 训练 → ONNX 部署）
- **报告生成**：大模型 API 将结构化结果转为自然语言
- **异步处理**：长文本分析走任务队列，避免阻塞

#### 验收标准

- 支持单次上传 ≥ 20,000 字
- 常见合同风险类型召回率 ≥ 85%（以标注集为准）
- 报告生成时间 ≤ 60 秒（10,000 字以内）
- 报告语言通过「去 AI 味」后处理检测

---

### 3.2 模块二：风格学习与写作

#### 功能描述

用户上传 3–5 篇参考文案（或聊天记录），系统学习其写作风格，再根据新需求生成**符合该风格**的内容。

#### 子功能

| 子功能 | 说明 |
|--------|------|
| 风格档案管理 | 创建、命名、更新、删除多个风格档案 |
| 样本上传与分析 | 支持 txt / docx / 粘贴，自动提取特征 |
| 风格预览 | 展示提取到的关键词、句长分布、语气词频率 |
| 定向生成 | 输入主题/大纲/关键词，选择风格档案后生成 |
| 一致性校验 | 生成后自动比对风格向量，不达标则重试 |
| 历史记录 | 保存每次生成的需求、结果、所用风格 |

#### 「去 AI 味」体现

- 保留用户惯用词（如「其实」「反正」「说白了」）
- 匹配句长偏好（短句党 vs 长句党）
- 匹配标点与分段习惯
- 避免强行升华、总结式结尾

#### 技术要点

1. **风格提取**：分词、去噪 → 词汇偏好、句长分布、修辞、语气词
2. **风格向量**：embedding 模型生成浮点向量，存入档案
3. **风格引导生成**：提示词注入风格描述 + 向量相似示例
4. **后处理**：相似度检测、禁用词过滤、口语化增强（可选）

#### 验收标准

- 3 篇样本即可创建可用风格档案
- 风格相似度低于阈值时自动重生成（最多 2 次）
- 用户可手动微调「正式 ↔ 口语」滑块
- 单次生成响应 ≤ 30 秒（流式输出首 token ≤ 3 秒）

---

### 3.3 模块三：智能聊天助手

#### 功能描述

帮助用户完成特定沟通场景（说服朋友打游戏、邀请聚餐、安慰、催稿、砍价等），AI 以**真人对话风格**生成可发送的消息，或模拟多轮对话演练。

#### 子功能

| 子功能 | 说明 |
|--------|------|
| 场景选择 | 预设场景模板 + 自定义目标 |
| 关系设定 | 朋友 / 同事 / 家人 / 客户 / 自定义 |
| 人设剧本 | 根据目标自动生成沟通策略与人格设定 |
| 多轮对话 | 支持用户扮演对方，AI 实时应对 |
| 消息建议 | 一键生成 3 条不同策略的回复候选 |
| 会话存档 | 保存人设、历史、结果评价 |

#### 技术要点

- Role Prompting + 场景剧本模板
- 情绪感知（基于对方最新一条消息的语气分类）
- 人格一致性约束（系统 prompt 中固化人设卡片）
- 流式输出（SSE / WebSocket）提升聊天体验

#### 验收标准

- 同一会话内人设不漂移（人工抽检）
- 支持至少 20 轮连续对话
- 每条回复 ≤ 150 字（可配置，模拟微信场景）
- 提供「太 AI 了，换一条」快捷重生成

### 4.2 后端（Django + DRF）

#### 目录结构（建议）

```
backend/
├── config/                 # 项目配置、URL 入口、Celery
├── apps/
│   ├── users/              # 用户、认证、偏好
│   ├── documents/          # 模块一：文案检索
│   ├── styles/             # 模块二：风格档案与生成
│   ├── chat/               # 模块三：聊天会话
│   ├── prompts/            # 提示词模板管理
│   └── tasks/              # 异步任务定义
├── core/
│   ├── llm/                # 大模型统一网关（多厂商适配）
│   ├── inference/          # ONNX 推理封装
│   ├── style/              # 风格提取与向量计算
│   └── deai/               # 去 AI 味检测与后处理
├── templates/prompts/      # Jinja2 提示词模板
└── manage.py
```

#### 关键设计

| 层级 | 职责 |
|------|------|
| API 层（DRF ViewSet） | 参数校验、权限、序列化、分页 |
| Service 层 | 业务编排，不直接操作 HTTP |
| Repository 层 | 数据库访问封装（可选，复杂查询时使用） |
| LLM 网关 | 统一 `complete()` / `stream()` 接口，厂商可插拔 |
| 提示词引擎 | Jinja2 渲染 + 版本选择 + A/B 分流 |

#### 异步任务

- **Celery + Redis**：文案检索、风格分析、批量生成
- 上传后立即返回 `task_id`，前端轮询或 WebSocket 推送进度
- 任务状态：`pending → running → success / failed`

#### 实时能力

- 聊天模块：**Django Channels + WebSocket**（或 SSE 作为轻量备选）
- 流式 token 推送至前端，模拟打字效果

### 4.4 数据库设计（云端 PostgreSQL）

> **部署策略**：主数据库采用**云端托管 PostgreSQL**，不自建、不随应用容器部署。具体云厂商（阿里云 RDS、腾讯云 PostgreSQL、Supabase、Neon、AWS RDS 等）**尚未选定**，应用层仅通过 `DATABASE_URL` 连接，与厂商解耦。

#### 4.4.1 云端数据库策略

| 维度 | 方案 |
|------|------|
| 引擎 | PostgreSQL 15+（必须支持 JSONB、UUID、GIN 索引） |
| 部署形态 | 云厂商托管实例（非 Docker 内自建） |
| 环境隔离 | **dev** / **staging** / **prod** 三套独立实例（或同实例不同 database） |
| 连接方式 | 统一 `DATABASE_URL` 环境变量；Django 通过 `dj-database-url` 或等价配置解析 |
| 传输安全 | 强制 SSL（`sslmode=require`）；生产禁用公网裸连，优先 VPC / 内网或 IP 白名单 |
| 连接池 | 应用侧使用 `pgbouncer` 或云厂商自带连接池（Serverless 实例通常内置） |
| 备份 | 依赖云厂商自动备份（建议 ≥ 7 天保留）+ 重大迁移前手动快照 |
| 迁移 | Django migrations 是唯一 Schema 变更入口；禁止手工改表 |

#### 4.4.2 云厂商选型参考（待定）

选型时按以下优先级评估，**MVP 阶段可先开 dev 实例，prod 厂商可后续再定**：

| 候选 | 适合场景 | 优点 | 注意点 |
|------|----------|------|--------|
| **阿里云 RDS PostgreSQL** | 国内部署、合规 | 生态成熟、与 ECS 内网互通、运维文档全 | 按规格计费，需配置白名单 |
| **腾讯云 PostgreSQL** | 国内部署 | 与 CVM 内网、控制台友好 | 部分高级特性因版本而异 |
| **Supabase** | 快速启动、海外/开发 | 免费 tier、自带 Dashboard、PostgreSQL 兼容 | 国内访问延迟需实测 |
| **Neon** | 开发/轻量 prod | Serverless、分支数据库（branch）适合 dev | 国内网络稳定性需评估 |
| **AWS RDS** | 出海或已有 AWS 账号 | 稳定、功能全 | 国内访问与合规成本高 |
| **Railway / Render** | 早期 MVP 一体化 | 开通快、与部署平台集成 | 规模上来后可能迁移 |

**硬性要求（无论选哪家）**：

- PostgreSQL ≥ 15，支持 JSONB
- 提供 SSL 连接串
- 支持自动备份与一键恢复
- dev 环境可接受公网 + IP 白名单；prod 必须内网或严格白名单

#### 4.4.3 本地开发如何连库

本地**不默认**跑 PostgreSQL 容器，而是：

1. **推荐**：连接云端 **dev 实例**（团队共享，数据一致）
2. **备选**：开发者离线时，可用 `docker-compose` 可选 profile 启动本地 PG（仅个人调试，Schema 仍以 migrations 为准）

```yaml
# docker-compose.dev.yml — Redis 必起；Postgres 仅 offline 备用
services:
  redis:
    image: redis:7
  # 可选：docker compose --profile offline up -d postgres
  postgres:
    profiles: ["offline"]
    image: postgres:16
```

#### 4.4.4 Schema 设计

所有主键使用 **UUID**，时间字段统一 UTC 存储。

#### 核心表

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| `users` | 用户账号 | email, password_hash, preferences (JSONB) |
| `style_profiles` | 风格档案 | user_id, name, style_vector (JSONB), features (JSONB), samples (TEXT[]) |
| `generation_records` | 生成记录 | user_id, style_id, prompt, result, model_name |
| `document_reviews` | 检索记录 | user_id, raw_text, risks (JSONB), report, model_version |
| `chat_sessions` | 聊天会话 | user_id, persona (JSONB), messages (JSONB), status, goal |
| `prompt_templates` | 提示词模板 | module, version, content, is_active, changelog |
| `async_tasks` | 异步任务 | task_id, type, status, result (JSONB), error |

#### 索引策略

- `user_id` 全表索引
- `chat_sessions(status, updated_at)` 复合索引
- `prompt_templates(module, is_active)` 复合索引
- JSONB 字段对高频查询 key 建 GIN 索引（按需）

### 4.5 AI 服务层

#### 4.5.1 提示词模板引擎（核心）

- **引擎**：Jinja2
- **分类**：`documents/`、`styles/`、`chat/`、`system/`
- **版本管理**：语义化版本 + Git 追踪 + 数据库 `is_active` 标记
- **A/B 测试**：按 user_id hash 分流模板版本
- **审核流程**：生产模板修改需双人 Review

**模板结构示例（概念）**：

```
[角色设定]
[风格约束] ← 动态注入 style_vector / features
[任务目标]
[对话历史] ← 聊天模块
[禁止事项] ← 禁用「赋能」「综上所述」等
[输出格式]
```

#### 4.5.2 大模型网关

统一接口：

```python
class LLMGateway:
    def complete(prompt, model=None, **kwargs) -> str: ...
    def stream(prompt, model=None, **kwargs) -> Iterator[str]: ...
```

支持厂商（可扩展）：

- OpenAI（GPT-4o / GPT-4）
- Anthropic（Claude）
- 百度文心、阿里通义（国内备选）

**调用流程**：

```
加载模板 → 注入参数 → 渲染 prompt → 调用 API → 后处理（去 AI 味 / 风格校验）→ 返回
```

#### 4.5.3 风格提取管线

```
样本文本
  → 清洗（去 HTML、多余空白）
  → 分词 & 特征统计（词频、句长、语气词、标点）
  → Embedding 模型 → style_vector (float[])
  → 写入 style_profiles
  → 生成时：向量 + 文本特征描述 双重注入 prompt
  → 生成后：cosine_similarity(generated, style_vector) ≥ threshold
```

#### 4.5.4 去 AI 味后处理

- **规则层**：黑名单词汇替换/警告（「赋能」「抓手」「在当今」等）
- **模型层**：轻量分类器判断「AI 味分数」
- **重试策略**：分数超阈值则换模板 temperature 或重生成

### 4.6 深度学习模型（模块一）

| 环节 | 方案 |
|------|------|
| 训练 | PyTorch |
| 部署 | ONNX → ONNX Runtime |
| 输入 | 文本分段 + 向量化 |
| 输出 | 风险类型 + 置信度 + 位置 span |
| 更新 | 版本号目录 + 蓝绿部署 + 灰度流量 |

### 4.7 基础设施与运维

#### 开发环境

```yaml
# docker-compose.dev.yml
services:
  redis:     redis:7          # 本地缓存 / Celery broker
  backend:   Django dev server  # DATABASE_URL 指向云端 dev 实例
  celery:    Celery worker
  frontend:  Vite dev (或本地 npm run dev)
  # postgres:  仅 --profile offline 时启用，作离线备用
```

- 数据库：连接**云端 dev 实例**（`.env` 中配置 `DATABASE_URL`）
- Redis：仍本地 Docker（或后续按需迁云）

#### 生产环境

- **容器化**：Docker Compose / K8s（规模扩大后）——**仅部署应用层，不含 PostgreSQL**
- **反向代理**：Nginx（静态资源 + API + WebSocket）
- **应用服务器**：Gunicorn + Uvicorn workers（ASGI for Channels）
- **数据库**：**云端托管 PostgreSQL**（prod 实例；高可用 / 只读副本按云厂商能力选配）
- **备份与恢复**：云厂商自动备份为主；应用层不执行 `pg_dump` 到容器内
- **监控**：Prometheus + Grafana；数据库指标接入云厂商监控或 postgres_exporter
- **日志**：结构化 JSON 日志 → ELK / Loki
- **模型追踪**：MLflow 记录版本与离线指标

### 4.8 安全

| 领域 | 措施 |
|------|------|
| 认证 | JWT + OAuth2（可选第三方登录） |
| 授权 | 路由级 + 对象级（只能访问自己的档案/会话） |
| 输入 | DRF Serializer 严格校验，文本长度上限 |
| 敏感信息 | 环境变量 / Secret Manager，不入库明文 |
| 数据库 | `DATABASE_URL` 仅环境变量注入；强制 SSL；prod IP/VPC 白名单 |
| CORS | 白名单域名 |
| 内容安全 | 用户输入过滤 + 生成结果敏感词检测 |

### 4.9 性能目标

| 指标 | 目标 |
|------|------|
| API P95 响应（非 AI） | < 200ms |
| 流式首 token | < 3s |
| 文案检索（1 万字） | < 60s |
| 并发用户（MVP） | 100 |
| 数据库查询 | 避免 N+1，热点走 Redis 缓存 |

### 11.1 名词解释

### 11.2 参考技术栈汇总

| 层级 | 选型 |
|------|------|
| 前端 | Vue 3, Vite, TypeScript, Pinia, Axios |
| 后端 | Django 5, DRF, Celery, Channels |

### 11.3 与原始需求的架构差异说明

| 项目 | 原始需求 | 本文档调整 | 理由 |
|------|----------|------------|------|
| 后端框架 | FastAPI | Django + DRF | Admin 后台、ORM 生态、权限体系更成熟，适合多模块 CRUD 密集场景 |
| 实时通信 | 提及 WebSocket | Django Channels 一等支持 | 与 Django 深度集成，聊天模块更自然 |
| 开发优先级 | 三模块并行 | 模块二 → 三 → 一 | 最快验证核心价值，降低对自研模型依赖 |
| 任务表 | 未单独列出 | 增加 `async_tasks` 表 | 统一异步任务状态管理，前端体验一致 |
| 数据库部署 | 本地 Docker / 自建主备 | 云端托管 PostgreSQL | 免运维、自动备份；厂商暂未选定，应用层与云解耦 |
